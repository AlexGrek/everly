# work (Everly)

## Workflow

1. Investigate the issue or request by reading documentation first, then code, then asking questions if needed. Find related skills for additional context. Use them.
2. Investigate if there are any tests related to that feature. If there are, read them to understand how the feature is supposed to work and what edge cases are covered.
3. Ensure you understand the problem and do not try to solve it before you do.
4. Think about the fix or feature implementation. If it is a large change, consider breaking it down into smaller steps or sub-tasks. Use TODO list. If it is significant change - use plan mode.
5. All architectural decisions should be made by the user, ask user questions if these are needed.
6. Implement the fix or feature, following best practices and coding standards. Write clean, maintainable code that is easy to understand. Split code by subfunctions and submodules. Ensure high code mainainability and readability.
7. Write tests for the new code, if applicable. Ensure that all existing tests still pass.
8. Document the changes you made, including any new features or bug fixes. Update any relevant documentation to reflect the changes.
9. Do a self-review. Check your code for any potential issues, such as bugs or performance problems. Ensure good readability. Do a refactoring step and re-run tests if needed.
10. Explain what you did to the user in very short summary. Include TL;DR section.

## Notes

Documentation is the source of truth, but it can be outdated or incomplete. Always check documentation first, but verify it against the code and ask questions if something is unclear or seems wrong.

When asking questions, be specific about what you don't understand and what you've already checked. This helps others provide better answers and also shows that you've made an effort to find the answer yourself.

Prefer asking quistions, do not assume you know the answer. It's better to ask and learn than to guess and potentially make a mistake.

## Architecture preferences

This is a game, so fast code is preferrable to good code. However, good code is better than bad code, so try to find a balance between the two. If you have to choose between writing clean code and writing code quickly, prioritize clean code. It will save you time in the long run.
